# v1.7 Federation Design — Team Antibody Federation

> Status: **COMPLETE — v1.7.0 RELEASE CANDIDATE** ✅
> Date: 2026-04-02 | Finalized: 2026-04-03

## 1. Problem Statement

When multiple repositories use `afd sync --remote <url>`, antibodies from different teams may share
the same local ID (e.g. `"missing-import-guard"`). Without namespacing, a blind `INSERT OR REPLACE`
would silently overwrite patterns from a different team. We need:

1. **Namespace isolation** — prevent cross-team ID collisions
2. **Versioning** — track mutation history per antibody
3. **Conflict resolution** — deterministic, zero-interaction policy for CI/CD environments

---

## 2. Namespace / Identifier Strategy

### Format: `<scope>/<name>`

| Component | Description | Example |
|---|---|---|
| `scope` | Team/org slug (lowercase, hyphenated) | `"myorg"` |
| `name` / `id` | Existing local antibody name | `"missing-import-guard"` |
| `fqid` | Globally unique: `"<scope>/<id>"` | `"myorg/missing-import-guard"` |

### Scope Resolution (`resolveScope()` in `src/core/federation.ts`)

Priority order:
1. `AFD_SCOPE` env var — explicit override for CI/CD pipelines
2. Git remote origin URL → parse org segment
   - SSH: `git@github.com:myorg/repo.git` → `"myorg"`
   - HTTPS: `https://github.com/myorg/repo.git` → `"myorg"`
3. Fallback: `"local"`

### Storage Encoding

- **Local antibodies** (`scope = "local"`): stored with original `id` (no change to existing rows)
- **Federated antibodies** (`scope != "local"`): stored with `fqid` as the `id` column value
  (`"myorg/missing-import-guard"`) to avoid PRIMARY KEY collisions without schema restructuring
- **Why not composite PK?** Requires full table migration; fqid-as-id achieves the same uniqueness
  guarantee with zero downtime and safe backward compatibility

## 3. Versioning Mechanism

### Monotonic Integer + ISO-8601 Timestamp

| Field | Type | Description |
|---|---|---|
| `version` | `number` | Monotonic integer, starts at 1, incremented on each content change |
| `updatedAt` | `string` | ISO 8601 — last mutation time (LWW tiebreaker) |
| `learnedAt` | `string` | Original learning time — **immutable** |

### Rationale

- **Monotonic int over semver**: Individual antibody patterns don't have API surfaces — semver is
  overkill and introduces ambiguity about what constitutes a "breaking" pattern change.
- **Monotonic int over timestamp-only**: Clocks can skew across machines. The integer is the
  authoritative comparison; timestamp is only used when integers are equal.
- **Monotonic int over hash**: A hash detects changes but doesn't express ordering. We need
  "which came later" not just "are they different".

## 4. Conflict Resolution Policy: RWIN (Remote-Wins-If-Newer)

### Decision Function (`remoteWins()` in `src/core/federation.ts`)

```
remote.version > local.version  →  Remote wins (take remote)
remote.version < local.version  →  Local wins (skip remote)
remote.version == local.version →  newer updatedAt wins
                                   on exact tie → Local wins (keep local)
```

### Behavior at `/antibodies/learn` endpoint

| Condition | Response |
|---|---|
| New antibody (not in DB) | Insert, status: `"learned"` |
| Remote is newer | Update in-place, status: `"updated"` |
| Local is newer or equal | No-op, status: `"skipped"` + reason |

### Why RWIN over interactive merge?

- afd is designed for CI/CD and background daemon contexts — no TTY assumed
- Interactive merge requires state across sessions (conflict markers in DB = complexity)
- RWIN with monotonic versioning is deterministic, auditable, and reproducible
- Teams that need manual review can use `--push` only (never `--pull` automatically)

## 5. Push Payload Stamping

When `afd sync --push --remote <url>` is called, the CLI:

1. Calls daemon `/sync` to flush local antibodies to `global-vaccine-payload.json`
2. Reads the payload and **stamps publisher scope** on all antibodies:
   ```
   ab.scope = publisherScope
   ab.fqid  = "<publisherScope>/<ab.id>"
   ab.version = ab.version ?? 1
   ab.updatedAt = ab.updatedAt ?? ab.learnedAt
   ```
3. POSTs the enriched payload to the remote URL

This ensures the remote server always receives fully-qualified antibodies regardless of whether
the local DB has been migrated yet.

## 6. Schema Changes (SQLite — `antibodies` table)

Three new columns added via safe `ALTER TABLE` migrations in `src/core/db.ts`:

```sql
ALTER TABLE antibodies ADD COLUMN scope     TEXT    NOT NULL DEFAULT 'local'
ALTER TABLE antibodies ADD COLUMN ab_version INTEGER NOT NULL DEFAULT 1
ALTER TABLE antibodies ADD COLUMN updated_at TEXT   NOT NULL DEFAULT (datetime('now'))
```

All migrations are wrapped in `try/catch` — safe to run on existing DBs.

## 7. TypeScript Interfaces

See `src/core/federation.ts`:

- `FederatedAntibody` — antibody with scope, fqid, version, updatedAt, learnedAt
- `FederatedPayload` — vaccine payload with publisher scope
- `resolveScope()` — scope derivation function
- `remoteWins()` — RWIN decision function

`src/commands/sync.ts` re-exports `FederatedAntibody` and `FederatedPayload` as `VaccineAntibody`
and `VaccinePayload` for backward compatibility with local push/pull functions.
